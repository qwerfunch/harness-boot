# harness-boot

**plan MD를 넣으면, 멀티 에이전트 하네스가 통째로 만들어지는 Claude Code 플러그인입니다.**

---

## 에이전트는 코드를 잘 짭니다. 다만...

Claude Code 에이전트는 똑똑합니다. 코드도 잘 쓰고, 시키면 대부분 해냅니다. 그런데 한동안 같이 일하다 보면 재미있는 패턴이 보입니다.

- 테스트를 슬쩍 건너뜁니다. 나름의 이유를 달면서요.
- 코드는 바꾸는데 문서는 잊습니다. 사람이랑 비슷하죠.
- 세션이 바뀌면 맥락이 날아가고, 같은 고민을 다시 합니다.

재미있는 건, 프롬프트에 "테스트 꼭 작성해"라고 적어도 에이전트는 왜 이번엔 안 해도 되는지 꽤 그럴듯한 이유를 찾아낸다는 겁니다. 규칙을 아는 것과 따르는 것은 다른 문제더라고요.

harness-boot은 여기서 출발합니다 — 규칙을 알려주는 대신, 구조로 자연스럽게 지켜지게 만들면 어떨까?

---

## 설계는 당신의 몫입니다

> 좋은 하네스는 결국 좋은 설계에서 나옵니다.
>
> 설계까지 자동화하면 편하겠지만, 적어도 지금은 설계와 실행을 한데 묶으면 둘 다 어중간해지는 경우가 많았습니다. 설계는 깎고, 부수고, 다시 쌓는 과정에서 단단해지는데 — 그걸 실행과 동시에 하면 충분히 깎이지 않더라고요.
>
> **그래서 harness-boot은 지금은 설계에 손대지 않습니다.** 아키텍처를 고민하고, 구조를 뒤엎고, 새벽까지 plan을 다듬는 건 당신의 영역입니다. 그 plan이 충분히 다듬어졌다고 느낄 때, 저희에게 넘겨주세요.
>
> 그다음은 저희가 합니다. 빈틈없이.

## harness-boot의 접근

### Plan 기반 맞춤 하네스 — 보일러플레이트가 아닙니다

같은 틀을 찍어내는 게 아니라, plan을 읽고 프로젝트에 맞게 하네스를 구성합니다. tech stack, architecture pattern, feature 목록을 plan에서 추출하고, 명시되지 않은 부분은 2-3개 선택지를 제안한 뒤 사용자가 고릅니다. 에이전트가 임의로 결정하지 않습니다.

### TDD 격리 — 서로 못 보게 하면 정직해집니다

테스트를 쓰는 에이전트와 구현하는 에이전트가 서로의 컨텍스트를 볼 수 없습니다. 구현 코드를 본 적 없는 에이전트가 테스트를 쓰니, 끼워 맞추는 일이 생기지 않습니다. 실패하는 테스트 작성(Red), 통과시키는 구현(Green), 정리(Refactor) — 각 단계가 별도 에이전트에서 실행됩니다.

### Domain Persona — 도메인을 이해하는 에이전트

`hashPassword`를 구현하는 에이전트는 "문자열을 받아서 해시를 리턴한다"만 알면 안 됩니다. SOC2 컴플라이언스 때문에 bcrypt cost 12 이상이 필수라는 맥락도 알아야 합니다. `/setup` 과정에서 harness-boot은 plan에서 도메인 페르소나를 추출합니다 — 프로젝트 목적, 핵심 엔티티, 비즈니스 규칙, 도메인 용어, 이해관계자 관심사. 그리고 각 에이전트에게 역할에 맞는 도메인 정보만 전달합니다. 오케스트레이터는 전체 그림을, 테스트 작성자는 현재 피처와 관련된 엔티티와 불변 규칙만 봅니다. 에이전트가 "부분적으로 맞지만 전체적으로 틀린" 결정을 내리는 일을 막습니다.

### Anti-Rationalization — 변명할 거 알고, 미리 답해뒀습니다

"리팩터링이라 테스트 안 해도 돼요", "내부 함수라 문서 안 써도 돼요" — 에이전트가 흔히 꺼내는 이유들이 있습니다. 모든 스킬에 이런 상황별 반박 테이블을 넣어뒀습니다. *"그렇게 말할 거 알아. 이번에도 해야 해. 이유는 이거야."* — 규칙을 반복하는 것보다 잘 통하더라고요.

### Runtime Hooks — 부탁 대신 구조로 지킵니다

프롬프트에 "문서도 같이 수정하세요"라고 써도, 바쁘면 잊을 수 있습니다. harness-boot은 hook으로 이걸 잡아둡니다 — 문서 변경 없이 코드만 커밋하려 하면 커밋이 진행되지 않습니다. 보안 위험이 있는 명령어도 같은 방식으로 걸러냅니다.

### Iteration Convergence — 끝없이 돌지 않습니다

피처 하나당 TDD 사이클은 최대 5회까지만 반복합니다. 그 안에 해결되지 않으면 에이전트가 혼자 계속 시도하는 대신, 상황을 정리해서 사용자에게 넘깁니다. 에이전트의 시간도, 사용자의 비용도 낭비하지 않기 위해서입니다.

### 세션 연속성 — 끊겨도 이어갑니다

세션이 바뀌어도 마지막 체크포인트에서 다시 시작합니다. 매 세션 시작 시 진행 상태와 피처 목록의 일관성을 자동으로 확인하고, 어긋나면 알려줍니다.

## 빠른 시작

```bash
# 1. 클론 후 Claude Code 플러그인으로 로드
git clone <this-repo>
cd harness-boot
claude --plugin-dir .

# 2. plan 투입
/setup path/to/plan.md

# 3. 각 생성 단계(1-6)를 확인하며 진행

# 4. 개발 시작 — 다음 피처를 골라 TDD 사이클을 돌립니다
/start
```

## 명령어

| 명령어 | 용도 | 사용 시점 |
|--------|------|-----------|
| `/setup <plan.md>` | plan을 읽고 하네스 구조 생성 | 프로젝트 시작 시 1회 |
| `/start` | 다음 피처를 골라 TDD 사이클 실행 | 하네스 생성 이후 반복 사용 |

## 생성되는 하네스 구조

```
project-root/
├── CLAUDE.md                  # 프로젝트 요약 (<1,500 토큰)
├── PROGRESS.md                # 진행 상태 + 체크포인트
├── feature-list.json          # 피처 목록 + 통과 여부
├── CHANGELOG.md               # 변경 이력
├── .claude/
│   ├── settings.json          # hook 설정
│   ├── agents/                # 에이전트 (모델 라우팅 포함)
│   ├── skills/                # 스킬 (Anthropic Agent Skills 포맷)
│   ├── protocols/             # TDD, iteration convergence, 문서 동기화 등
│   ├── examples/              # 참고 샘플
│   ├── domain-persona.md      # 에이전트용 도메인 컨텍스트
│   └── *.md                   # context-map, security, quality-gates 등
├── hooks/                     # hook 스크립트
└── scripts/                   # 자동화 유틸리티
```

## 마치며

오랫동안 개발하면서 반복적으로 부딪혔던 문제들이 있습니다. 테스트를 미루고, 문서를 잊고, "이번엔 괜찮겠지"라고 넘기는 것. 사람에게는 문화로 풀었던 것을, 에이전트에게는 구조로 풀었습니다. harness-boot은 그 경험에서 나왔습니다.
