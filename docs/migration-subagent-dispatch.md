# Migrating a Generated Harness to Subagent Dispatch

Applies to harnesses generated by `/harness-boot:setup` **before** the Subagent Dispatch refactor (PRs #31â€“#34). If you ran `/setup` more recently, your harness already uses the new model â€” no action needed.

## What changed

The generated harness previously relied on experimental Agent Team primitives (`TeamCreate`, `SendMessage`, `TaskCreate`, `TaskUpdate`) that were gated behind `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`. Without that flag, the runtime silently fell back to single-agent execution, so the orchestrator's multi-module plan was effectively inert.

The refactor replaces that path with **Subagent Dispatch**:

- The orchestrator dispatches work by emitting multiple `Agent(subagent_type=<slug>, prompt=...)` tool_use blocks in a single response (Claude Code's canonical parallel pattern).
- Coordination is turn-based through file envelopes at `_workspace/handoff/{from}->{to}.md` â€” no live messaging between simultaneously-running subagents.
- `PROGRESS.md` remains the single per-feature progress store; `_workspace/` holds per-phase artifacts.

See `docs/references/execution-model-divergence.md` for the full rationale and pattern-preservation mapping.

## Does this affect me?

Run this in your project root:

```bash
grep -rE 'TeamCreate|SendMessage|TaskCreate|TaskUpdate' .claude/ 2>/dev/null | grep -v '(legacy|previously|formerly)'
```

Hits outside explanatory prose mean your generated harness still references the old primitives and should be migrated.

You should also migrate if any agent file at `.claude/agents/*.md` contains a `## Team Communication Protocol` section â€” this is replaced by `## Handoff Protocol` under the new model.

## Migration path A â€” Regenerate (recommended)

If you have not hand-edited the generated `.claude/` beyond what `/setup` produced, regeneration is the safest path:

1. Preserve your plan MD and any customizations outside `.claude/` (your `src/`, tests, `feature-list.json` progress, `PROGRESS.md`, `CHANGELOG.md`).
2. Re-run `/harness-boot:setup <plan.md>`. When prompted about existing files, answer that you want to overwrite the harness scaffolding; confirm you want to keep `feature-list.json` `passes` flags and `PROGRESS.md` progress state.
3. Verify: `scripts/verify-subagent-dispatch.mjs` (ships with this plugin â€” run it against the target project root).

Regeneration rewrites all agent bodies, protocols, and rules from the current templates â€” the result is indistinguishable from a fresh install at the new version.

## Migration path B â€” Manual patch (if customized)

If you have hand-edited generated agent bodies and cannot regenerate, apply these substitutions:

1. **Rename section headers in every `.claude/agents/*.md` that has one**:
   - `## Team Communication Protocol` â†’ `## Handoff Protocol`
2. **Replace the message-format rule fragment body**. In each agent body that contains the core-fields table (orchestrator, implementer-<slug>, reviewer, qa-agent), replace its core-fields section with the current version at `docs/templates/agents/rules/06-message-format.md` â€” the table now documents handoff envelopes at `_workspace/handoff/{from}->{to}.md` rather than `SendMessage` payloads.
3. **Rewrite the orchestrator's dispatch body**. The old `TeamCreate(...) â†’ TaskCreate(...) â†’ SendMessage` flow is replaced by parallel `Agent(subagent_type="implementer-<slug>", prompt=...)` tool_use blocks. Use the current `commands/start.md` "Parallel dispatch (Subagent Dispatch model)" section as the template.
4. **Remove `task-assigned` / `cancel-pending` message kinds** from any custom rule fragments â€” the former becomes the `Agent` tool prompt itself; the latter has no equivalent under Subagent Dispatch (stop is wave-boundary, not live).
5. **Protocols**: replace `.claude/protocols/message-format.md` body with the current `docs/templates/protocols/message-format.md`. The iteration-cycle protocol only needs a one-line fix in its Concurrency section (see `docs/templates/protocols/iteration-cycle.md`).

Re-run the verification script after patching.

## What does not change

- **Hooks**. `pre-tool-coverage-gate.mjs`, `pre-tool-doc-sync-check.mjs`, and the PR #31 staged-vs-HEAD flip-detection fix are unchanged. Hooks run on `git commit` (Bash tool invocation), not on Agent dispatch, so the Subagent Dispatch transition does not affect them.
- **Quality Gates 0â€“5**. Gate logic and evidence requirements are the same; the gate runs inside the dispatched implementer or reviewer, just as before.
- **TDD / BDD sub-agent isolation**. `tdd-test-writer` / `bdd-writer` / `tdd-implementer` / `tdd-refactorer` were already called via the `Agent` tool before the refactor; they are unaffected.
- **`feature-list.json` schema**, `PROGRESS.md` schema, `CHANGELOG.md` format, `scripts/update-feature-status.mjs`.
- **Slash command interface**. `/harness-boot:setup` and `/harness-boot:start` take the same inputs.

## Verification checklist

After regeneration or patching:

1. `scripts/verify-subagent-dispatch.mjs` returns a clean report.
2. `.claude/agents/orchestrator.md` describes dispatch via parallel `Agent` tool_use blocks and reads handoff envelopes from `_workspace/handoff/`.
3. Each `.claude/agents/implementer-<slug>.md` contains a `## Handoff Protocol` section; `architect.md`, `debugger.md`, `tester.md`, `bdd-writer.md`, and all `tdd-*.md` do **not**.
4. A dry run of `/harness-boot:start` on an open feature produces `_workspace/02_impl_<slug>_<feature>.md` and, at Gate 2, `_workspace/handoff/implementer-<slug>->reviewer.md`.

If any step fails, open an issue with the output of step 1 attached.
