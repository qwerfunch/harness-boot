# v0.4 Milestone Plan — Agent orchestration & interactive flows

**Status**: draft · 2026-04-23 작성 · v0.3.13 이후 다음 minor bump

## 1. Rationale — 왜 minor bump

`patch-first` 정책 유지. v0.4 는 다음 셋이 **같이** 움직여야 의미가 생기는 agent/automation 레이어 묶음이라 minor:

1. **F-012 Claude sub-agents** — Tool 권한 매트릭스 + plugin-ships `agents/`
2. **F-014 Hooks 시스템** — 6 core hooks + plugin-ships `hooks/` + fail-open enforcement
3. **F-002 Modes A/R/B-2** — spec.yaml 의 LLM-드리븐 editing 경로 실구현

셋 다 "agent 가 도구를 통해 사용자 의도를 실행" 이라는 단일 스토리. 분리 patch 로 풀면 3 번의 의미 없는 bump 가 생기고 사용자는 전체 시나리오를 경험 못 함. v0.3.13 state.yaml 이 86% done (19/22) 이고 남은 3 개가 정확히 이 묶음 → v0.4 로 완결.

## 2. Scope — included / excluded

### Included (v0.4.0 릴리즈 대상)
- F-012 sub-agents (+ `agents/` 디렉터리 인프라)
- F-014 hooks (+ `hooks/hooks.json` + fail-open 헬퍼)

- Phase 3 CI (.github/workflows/) — v0.4 에 묶으면 커지므로 v0.5 후보

## 4. Per-feature design

### 4.1 F-023 `agents/` 인프라 (신규, pre-F-012)
- 디렉터리: `agents/` at plugin root
- 파일: `agents/<name>.md` — 각 에이전트의 system prompt + frontmatter
- frontmatter: `name`, `description`, `tools` (allow-list), `model` (optional)
- plugin.json 이 `agents/` 를 자동 스캔 또는 명시 선언 (Claude Code 규약 확인 필요)
- **AC**: `agents/` 디렉터리 tracked · plugin install 후 Claude Code 가 `/task <agent>` 노출

### 4.2 F-012 Sub-agents — Tool 권한 매트릭스
- 3 core agents 최소:
  - **orchestrator** — 상위 조율, 모든 tool 허용
  - **implementer** — 구현자, Edit/Write/Bash 허용 · Git push 금지
  - **reviewer** — 리뷰어, Read/Grep/Bash(read-only) · Edit 금지
- 권한 매트릭스 표 (agents/README.md)
- `commands/work.md` 가 implementer 에게 delegate, reviewer 가 후속 검증
- **AC-1**: 각 에이전트 tools 선언이 spec 과 일치
- **AC-2**: 권한 외 tool 요청 시 Claude Code 차단 (Claude Code 자체 enforcement 확인)

### 4.3 F-024 `hooks/` 인프라 (신규, pre-F-014)
- 파일: `hooks/hooks.json` — 선언형 훅 등록
- 구조:
  ```json
  {
    "hooks": {
      "PreToolUse": [{ "matcher": "Bash", "hooks": [...] }],
      ...
    }
  }
  ```
- 각 훅은 bash 커맨드 or agent/prompt hook
- plugin 설치 시 Claude Code 가 이 파일 읽음 (Claude Code 규약 확인 · update-config skill 문서 참고)
- **AC**: `hooks/` 디렉터리 tracked · 세션 시작 시 훅 로드 확인 가능

### 4.4 F-014 Hooks — 6 core + fail-open
- 6 core hooks:
  1. **PreToolUse Bash** → security-gate (rm -rf / sudo 패턴 감지 → `echo warn; exit 0` fail-open)
  2. **PostToolUse Write|Edit** → format (prettier/black 있으면 실행, 없으면 skip · `|| true`)
  3. **PostToolUse Write|Edit** → doc-sync-check (CLAUDE.md @import 여전히 resolve 하는지 grep)
  4. **PostToolUse Bash** → test-runner (opt-in via harness.yaml flag · 기본 비활성)
  5. **SessionStart** → session-bootstrap (banner + /harness:status 요약 추천)
  6. **PreToolUse Bash(rm|mv -r)** → coverage-gate-style (대량 삭제 사전 경고)
- 전부 **fail-open**: 훅 스크립트 exit != 0 여도 Claude Code 명령 차단 금지 (`|| true` 또는 exit 0 강제)
- 테스트: 각 훅을 direct execution 으로 JSON payload 파이프 테스트 (update-config skill §Constructing a Hook with verification 절차 참조)
- **AC-1**: 6 훅 전부 등록 · pipe-test PASS
- **AC-2**: 훅 의도적 실패 시나리오에서 Claude Code 명령은 계속 작동 (fail-open 검증)

### 4.5 F-002 Modes A/R/B-2 LLM 루프
- 현재 상태: Mode classifier (spec_mode_classifier.py) + Mode E (explain_spec.py) + diff renderer (spec_diff.py) shipped. Modes A/R/B-2 의 실제 대화 흐름은 commands/spec.md 에 prose 로만.
- 구현:
  - **Mode A (addition)**: classifier 가 A 로 분기 후 LLM 이 `<existing spec>` + user intent 를 읽어 diff 제안 → spec_diff.py 로 사용자 리뷰 → 사용자 approve 시 Edit tool 로 적용
  - **Mode R (refine)**: spec 읽고 "정합 개선" 제안 (중복 필드 병합, BR-NNN 형식 통일, description 공란 채움 등) → diff 제안 → approve
  - **Mode B-2 (plan.md 변환)**: `/harness:spec --from plan.md` → skills/spec-conversion 호출 (skill 이 이미 v0.5 shipped) → 결과 spec.yaml 쓰기
- 각 Mode 는 commands/spec.md 에 **재현 가능한 prompt template + approval checkpoint + mtime 보호** 세 축
- **AC-1**: 각 Mode 가 결정론적 분기 (same input → same mode) · 이미 classifier 구현됨
- **AC-2**: A/R 이 git diff 로 사용자 리뷰 가능 + `--dry-run` 지원
- **AC-3**: 모드 선택 결과가 preamble 의 mode 행에 표시

## 6. Risks

### R1. Claude Code 의 plugin agents/hooks 규약 미확정
- `agents/` 가 plugin 에서 자동 스캔되는지 확증 안 됨
- `hooks.json` 경로가 `.claude-plugin/` 하위인지 plugin-root 인지 불명
- **대응**: F-023 F-024 의 첫 작업이 Claude Code 2.1.x 매뉴얼 재확인 (claude-code-guide 에이전트 호출 권장)

### R2. Plugin install 시 agents/hooks 가 사용자 기기로 복사되는 부작용
- 플러그인 설치 = git clone 이므로 agents/hooks 도 사용자 기기로 복사됨
- 사용자가 원하지 않는 훅이 강제 발동 → 사용자 경험 저해
- **대응**: 모든 훅은 **opt-in** (harness.yaml 플래그 기본 off) · agents 는 user-facing slash 로 exposed (그래서 영향 관리)

### R3. F-002 LLM 루프는 자동 테스트 불가
- Mode A/R 의 제안 품질은 사람이 판단
- 회귀 감지 방법: golden spec 샘플 + classifier 결과 고정 + diff 형식 pinned
- **대응**: unit test 는 classifier + diff helper 에만, LLM 흐름은 commands/spec.md 의 prose contract + 수동 스모크
