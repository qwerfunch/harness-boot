# Retro Jumper — 2D 점프 게임 기획안

**작성자**: 게임 PM  
**작성일**: 2026-04-20  
**버전**: 0.2 (draft)  
**상태**: 프로토타입 판단용

---

## 1. 배경 / 컨셉

사내 해커톤에서 팀원들과 간단한 레트로 게임을 만들어보자는 아이디어. 무한 러너 + 점프가 메인 루프. 타겟은 웹에서 10분 안에 "한 판 해봤네" 경험을 주는 것. 장기적으론 모바일 웹/브라우저에서 leaderboard 공유까지. 상용 목적은 아니지만 브랜딩용으로 써도 될 만한 퀄리티를 노림.

영감: Jetpack Joyride, Crossy Road 의 간결함 + 70년대 도트 픽셀 아트.

## 2. 타겟 플레이어

- 자투리 시간을 때우는 일반 웹 방문자. 가입 없이 즉시 플레이.
- 20~40대, 게임 라이트 유저.
- 입력: 스페이스바(데스크톱), 화면 탭(모바일).

## 3. 게임 플레이 / 핵심 루프

### 3.1 주 루프 (필수)

- 캐릭터는 화면 좌측 고정, 자동 전진 (2D side-scroller)
- 장애물(바위, 구덩이)이 우측에서 다가옴
- 점프 버튼으로 회피. 미스 시 즉사·게임 오버
- 난이도: 시간 경과에 따라 스크롤 속도 증가 (초당 10%씩 가속? — 튜닝 필요)
- 점수: 1초당 10점 + 수집 아이템당 50점

### 3.2 수집 아이템 (필수)

- "코인" 에셋. 공중에 떠다니며 점프 궤적 위에 배치
- 수집 시 점수 + 효과음

### 3.3 파워업 (있으면 좋음)

- 일시 무적(3초), 2x 점수, 슬로우모션 등
- 일단 v1 은 무적 하나만

### 3.4 리더보드 (필수)

- 게임 종료 시 점수 + 닉네임 3자 입력 (옛날 오락실 스타일)
- Top 10 공개 / 본인 점수 개인 기록
- 닉네임 욕설 필터링 — 기본 수준만

### 3.5 설정 (nice-to-have)

- 음소거, 픽셀 배율(1x/2x/3x), 키 리매핑 (v1 범위 밖)

## 4. 아트 / 사운드

- 해상도: 256x144 가상 해상도, 디스플레이는 배율 스케일
- 팔레트: 16색 고정 (Pico-8 스타일)
- 스프라이트: 캐릭터 8x8, 장애물 8x16, 배경 타일 8x8
- BGM: 8-bit 루프. chiptune. 30초 이내 루프.
- SFX: 점프, 수집, 게임오버 3종 최소.

디자이너가 에셋 초안 1세트 제공. 추가 스프라이트는 필요시 요청.

## 5. 기술 스택

- **렌더링**: HTML5 Canvas + requestAnimationFrame (WebGL까지 안 써도 됨)
- **언어**: TypeScript
- **번들러**: Vite
- **프레임워크**: **미결정** — Phaser 3 쓸지, 직접 엔진 짤지 논의 중. 직접 짜는 편이 학습 가치 높지만 일정 리스크. (주니어 2명이 참여하는 프로젝트)
- **배포**: Cloudflare Pages (정적 호스팅)
- **서버**: 리더보드용 초경량 API — Cloudflare Workers + D1 (SQLite)

> Unity/Godot 쓰자는 의견도 있었지만 웹 배포 편의성·번들 크기 때문에 제외.

## 6. 비기능 요구사항

- **프레임레이트**: 60 FPS 유지 (저사양 브라우저에서도 30 FPS 보장)
- **로딩 시간**: 초기 로드 2초 이내 (에셋 합산 500KB 이내 목표)
- **메모리**: 장시간 플레이 시 누수 없음 — 10분 이상 플레이 메모리 증가 10MB 미만
- **입력 지연**: 점프 입력 → 시각적 반응 16ms 이내 (1 frame)

## 7. 리더보드 / 서버

### 7.1 데이터 모델

**Score**
- id, nickname (3자), score, played_at, client_hash (중복 제출 방지)

**AbuseLog** (자동 차단 기록)
- id, ip, reason, logged_at

### 7.2 API

- `POST /api/scores` — {nickname, score, play_duration, client_hash}
- `GET /api/scores/top?n=10`
- `GET /api/scores/around?score=N&window=5` — 내 점수 근처 10건

### 7.3 부정 방지

- 1플레이 당 최소 소요 시간 계산 (score ÷ 10 = 예상 플레이 초) — 이보다 짧으면 거부
- 같은 client_hash에서 1분에 2회 이상 제출 시 차단
- 리플레이 기록 저장은 v1 범위 밖 (v2 검토)

## 8. UI / 화면

- 타이틀 화면 — 게임 시작, 리더보드 보기, 설정
- 게임 플레이 — 점수 HUD 상단
- 게임 오버 — 점수 표시, 닉네임 입력 / 재시작
- 리더보드 — Top 10 + 본인 기록
- 설정 (음소거만)

## 9. 엣지 케이스 / 예외

- 창 비활성화(탭 전환) 시 게임 정지 + 복귀 시 재개 (아니면 게임 오버?) — **미결정**
- 화면 크기 변경 시 리사이즈 처리 (단, 플레이 중엔 고정)
- 모바일 방향 변경: 가로 모드 강제 (세로는 지원 안 함)
- 네트워크 없을 때: 로컬 플레이는 가능, 점수 제출 실패 시 로컬에만 저장 (다음 온라인 시 재시도? — 일단 안 함)
- 음소거 설정 지속: 세션 내에서만 (localStorage 로 영속화할지 — v1 범위 밖)

## 10. 외부 의존성

- Cloudflare Workers (리더보드 API)
- Cloudflare D1 (스토어)
- (선택) Cloudflare Analytics (DAU/세션 통계)
- 에셋 자체 제작 — 외부 라이선스 없음

## 11. 성공 지표

- 해커톤 발표 직후 3일간 총 플레이 세션 수 > 500
- 평균 세션당 플레이 횟수 > 3판
- 리더보드 제출율 > 40% (제출 UX 장벽 낮음 지표)

## 12. 일정

- **M1 (1주)**: 프로토타입 — 캐릭터 움직임, 장애물 1종, 점프, 게임 오버
- **M2 (1주)**: 난이도 곡선, 아이템, SFX
- **M3 (3일)**: 리더보드 서버 + 통합
- **M4 (2일)**: 폴리싱, 버그 수정, 발표 준비

총 2.5주 타이트. 해커톤 데모 일정.

## 13. 위험 / 가정

- 주니어 2명이 메인. 프레임워크(Phaser) 미경험. 직접 짤 경우 학습 시간이 일정 안에 맞을지 가정이 큼.
- 디자이너 투입 가능 가정 (0.3 FTE, 1주). 미확정.
- 리더보드 부정 방지는 "기본만" — 진지한 공격 시 뚫릴 수 있음. 해커톤 데모 수준에서는 충분.
- 30 FPS 하한이 저사양 안드로이드 크롬에서 정말 유지되는지 — 실기 테스트 필요.

## 14. 오픈 이슈

1. Phaser 3 vs 직접 엔진 — 주니어 학습 가치 vs 일정 안전성
2. 탭 비활성화 처리 (정지 vs 게임 오버)
3. 리더보드 닉네임 욕설 필터링 수준 (기본 사전 vs 수동 승인 큐)
4. 점수 제출 실패 시 재시도 정책
5. 파워업 확장 범위 (v1에 무적만? 아니면 3종까지?)
